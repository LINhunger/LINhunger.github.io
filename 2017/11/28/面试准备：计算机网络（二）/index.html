一、TCP协议

（一）概述

  1. TCP协议也叫传输控制协议，它提供了一种面向连接的、可靠的字节流服务。
  2. TCP位于传输层，属于端到端的协议。

（二）TCP如何实现可靠性

  1. 应用数据会被分割成TCP认为最合适发送的数据块。
  2. 当TCP发送一个段后，它启动一个定时器，等待目的端确认收到这个报文段。如果不能及时确认收到这个报文段，将会进行重发。
  3. 当TCP收到发自TCP连接另一端的数据，它将发送一个确认。
  4. TCP将保持它首部和数据的校验和。如果收到的报文段有差错，TCP将丢弃这个报文段和不确认收到该报文（希望超时重发）。
  5. TCP报文段使用IP数据包在网络层传输，而IP数据报的到达可能会失序，因此TCP报文段的到达也可能会失序。如果必要，TCP将对收到的数据进行重新排序，将收到的数据以正确的顺序交给应用层。
  6. 因为重传机制的存在，IP数据包会发生重复，TCP接收端会丢弃重复的数据。
  7. TCP还可以提供流量控制。TCP连接的每一方都有固定大小的缓冲空间。TCP的接收端只允许另一端发送接收端缓冲区所能接纳的数据。这防止了较快主机致使较慢主机的缓冲区溢出。

（三）三次握手

下面是建立一条TCP连接的过程：

  1. 请求端（通常称为客户端）发送一个SYN段指明客户端打算连接的服务器的端口，以及初始序号（ISN=x）。这个SYN段为报文段1。（seq : Sequence Number）
  2. 服务器发回包含服务器的初始序号的SYN报文段（报文段2）作为应答。同时，将确认序号设置为客户的ISN加1（ack = x+1）以对客户端的SYN报文段进行确认。一个SYN将占用一个序号。
  3. 客户必须将确认序号设置为服务器的ISN+1以对服务器的SYN报文段进行确认（报文段3）。

（四）四次挥手

下面是TCP连接断开的过程：

  1. 有应用层客户端主动发送断开命令后，TCP客户端发送一个FIN（用来关闭客户端到服务器的数据传送）。
  2. 当服务器收到这个FIN，它会发回一个ACK，确认序号为收到的序号加1。和SYN一样，一个FIN将占用一个序号。同时TCP服务器还向应用程序（即丢弃服务器）传用一个文件结束符。
  3. 接着这个服务器程序就关闭它的连接，导致它的TCP端发送一个FIN。
  4. 客户端收到后，发送一个确认，并将确认序号设置为收到的序号加1。

（五）为什么握手三次而挥手四次

建立一个连接需要三次握手，而终止一个链接需要经过四次挥手。

  1. TCP连接是全双工通信，因此每个方向需要进行单独的关闭。
  2. 收到一个FIN意味着在一个方向上没有了数据的流动。因此第一次服务端收到FIN后，就会通知应用层去关闭相应的连接，同时立即发送一个确认包给客户端。（这中间通知上层关闭连接需要一些准备工作。）
  3. 此时服务端仍然可以发送数据给客户端，直到服务器发送FIN给客户端。
